**Версия:** 0.1 (черновик)
**Дата создания:** 2026-06-13
**Статус:** живой документ — дополняется и уточняется по ходу проекта

## 1. Общие сведения

### 1.1 О компании

Компания занимается IT-аутсорсингом и системным администрированием для двух категорий клиентов:

- **Офлайн-клиенты** — обслуживание физической инфраструктуры: ПК, ноутбуки, серверы, сетевое оборудование (роутеры, свитчи, точки доступа), IP-телефония/АТС, периферия (принтеры, МФУ, ИБП) и др.
- **Облачные клиенты** — обслуживание облачной инфраструктуры: удалённые рабочие столы (RDP/терминальные серверы), 1С в облаке, виртуальные машины и др.

### 1.2 Назначение системы

Корпоративный портал — единая платформа для:

- регистрации и обработки заявок от клиентов (service desk);
- учёта обслуживаемого оборудования и ИТ-активов (CMDB);
- хранения и использования базы знаний (внутренней и внешней);
- AI-ассистента, помогающего клиентам, инженерам и руководству;
- аналитики по заявкам, затраченному времени и нагрузке.

### 1.3 Цели проекта

- Дать единый инструмент учёта заявок и оборудования вместо разрозненных таблиц/систем
- Снизить время реакции и решения заявок за счёт AI-ассистента
- Дать клиентам прозрачность: статус заявок, своё оборудование, база знаний
- Дать руководству аналитику по загрузке, SLA и списанному времени (основа для биллинга)

## 2. Пользователи и роли

### 2.1 Клиент

- Создаёт заявки (портал; в перспективе — email/Telegram/телефон)
- Видит статус и историю своих заявок
- Видит свой инвентарь оборудования (офлайн) / ресурсов (облако)
- Доступ к публичной части базы знаний
- Может общаться с AI чат-ботом

### 2.2 Инженер (исполнитель)

- Получает назначенные заявки, обновляет статус, комментирует
- Фиксирует затраченное время по заявке
- Использует AI-помощника: анализ заявки, похожие кейсы, предложенное решение, черновик ответа клиенту
- Обновляет данные об оборудовании в CMDB при визитах/работах

### 2.3 Руководитель / диспетчер

- Видит все заявки, распределяет и переназначает исполнителей
- Настраивает SLA, приоритеты, правила авто-роутинга
- Просматривает аналитику: загрузка инженеров, время по клиентам, SLA-нарушения
- Делает запросы к AI-аналитике на естественном языке

### 2.4 Администратор системы

- Управляет пользователями, ролями и правами доступа
- Настраивает справочники: типы оборудования, категории заявок, SLA-политики
- Настраивает интеграции (база знаний, AI, телефония и др.)

## 3. Функциональные требования

### 3.1 Модуль заявок (Service Desk)

Базовый функционал, аналогичный okdesk/GLPI:

- **Создание заявки**: тема, описание, тип/категория, приоритет, вложения, привязка к оборудованию (опционально), привязка к контактному лицу клиента
- **Каналы создания**: портал (MVP); email, Telegram, телефония — последующие этапы (см. раздел 7 "Открытые вопросы")
- **Жизненный цикл**: настраиваемые статусы (например: Новая → В работе → Ожидание ответа клиента → Решена → Закрыта)
- **Назначение**: на инженера или группу — вручную и автоматически (через AI-роутинг)
- **SLA**: настраиваемые сроки реакции/решения по приоритету и/или клиенту, индикация нарушений
- **Комментарии**: внутренние (видны только инженерам) и внешние (видны клиенту)
- **Учёт времени**: фиксация затраченного времени на каждую заявку/действие, привязка к биллингу
- **История изменений**: полный аудит-лог по заявке
- **Связь с CMDB**: заявка может быть привязана к конкретной единице оборудования

### 3.2 Модуль инвентаризации оборудования (CMDB)

- **Учёт активов**:
  - Офлайн: ПК, ноутбуки, серверы, мониторы, сетевое оборудование, IP-телефония/АТС, периферия (принтеры, МФУ, ИБП) и т.д.
  - Облачные: виртуальные машины/RDP-серверы, базы 1С, объём ресурсов (CPU/RAM/диск), лицензии
- **Карточка актива**: тип, модель, серийный номер, дата ввода в эксплуатацию, гарантия, текущее состояние, ответственное лицо, локация (адрес/кабинет клиента — для офлайн)
- **Привязка к клиенту**: каждый актив принадлежит клиенту (опционально — конкретной локации/подразделению)
- **История обслуживания**: связь с заявками, в которых упоминался/обслуживался актив
- **Учёт лицензий ПО**: срок действия, количество мест
- **Импорт/экспорт**: массовый импорт оборудования (CSV/Excel) при первоначальном наполнении базы

### 3.3 База знаний

- **Внутренняя база знаний**: статьи для инженеров (типовые решения, регламенты, особенности клиентов)
- **Публичная часть**: статьи, видимые клиентам (инструкции, FAQ)
- **Подключение внешней базы знаний**: интеграция с внешними источниками (Confluence/Notion/Wiki — конкретный сервис уточняется), чтобы AI мог использовать существующие материалы без переноса
- **Использование AI**: статьи индексируются и используются как контекст для AI-ассистента (RAG)

### 3.4 AI-ассистент (killer-фича)

Согласованы 4 приоритетных сценария:

1. **Помощник инженера по заявке**
   - При открытии заявки AI анализирует описание проблемы
   - Ищет похожие закрытые заявки и релевантные статьи базы знаний
   - Предлагает вероятную диагностику и шаги решения
   - Помогает сформулировать ответ клиенту (черновик)

2. **Чат-бот для клиентов**
   - Клиент описывает проблему (портал, в перспективе — Telegram)
   - AI задаёт уточняющие вопросы и формирует корректно оформленную заявку (категория, приоритет, описание)
   - По типовым вопросам может дать готовый ответ из базы знаний без эскалации на инженера

3. **Авто-классификация и роутинг заявок**
   - При создании заявки AI определяет категорию и приоритет
   - Назначает заявку на нужного специалиста/группу с учётом компетенций, загрузки и истории клиента

4. **AI-аналитика на естественном языке**
   - Запросы вида «сколько времени потрачено на клиента X за квартал», «какие проблемы повторяются чаще у клиента Y», «загрузка инженеров за месяц»
   - AI формирует агрегации по данным портала и возвращает ответ (и визуализацию, где применимо)

**Архитектурное требование:** AI-ассистент строится как отдельный сервис/модуль с доступом к данным портала (заявки, CMDB, база знаний) через контролируемый внутренний API. Это закладывается в архитектуру с самого начала, даже если сценарии реализуются поэтапно (см. раздел 6 "Roadmap").

### 3.5 Аналитика и учёт времени

- **Учёт времени**: тайм-трекинг по заявкам (вручную или таймером), привязка к биллингу клиента
- **Отчёты**:
  - По заявкам: количество, время решения, соответствие SLA — по категориям/клиентам/инженерам
  - По времени: списанные часы по клиентам/проектам за период — основа для биллинга
  - По загрузке: распределение нагрузки между инженерами
- **Дашборды**: сводная картина для руководителя (открытые заявки, нарушения SLA, топ-клиенты по времени и т.д.)
- **Экспорт**: отчёты в Excel/PDF

## 4. Нефункциональные требования

- **Язык интерфейса**: русский (основной); архитектура должна поддерживать локализацию
- **Безопасность**:
  - Ролевая модель доступа (RBAC)
  - Изоляция данных между клиентами (клиент видит только свои заявки и оборудование)
  - Защита персональных и конфиденциальных данных клиентов
- **Производительность**: отклик интерфейса в пределах 1-2 сек на типовых операциях (точные требования — после уточнения масштаба, см. раздел 7)
- **Доступность**: резервное копирование БД и план восстановления (детали — после получения информации о сервере)
- **Расширяемость**: модульная архитектура, чтобы новые требования добавлялись без переделки ядра

## 5. Технологическая архитектура

- **Backend**: NestJS (Node.js/TypeScript)
- **Frontend**: Next.js + React, TypeScript
- **База данных**: PostgreSQL
- **AI**: Claude API (Anthropic) — отдельный AI-сервис с RAG над базой знаний и данными портала
- **Развёртывание**: Docker / Docker Compose, целевая платформа — собственный сервер компании (детали ожидаются)
- **Репозиторий**: GitHub (доступ будет предоставлен)

## 6. Этапы разработки (Roadmap, предварительно)

### MVP (этап 1)

- Модуль заявок: создание, статусы, назначение, комментарии, учёт времени
- Базовая CMDB: карточки оборудования, привязка к клиентам
- Внутренняя база знаний (минимальный редактор статей)
- Роли: клиент, инженер, руководитель/диспетчер, админ
- AI: помощник инженера по заявке (анализ + похожие кейсы + черновик ответа) — первый AI-сценарий, наиболее ценный и наименее рискованный

### Этап 2

- Авто-классификация и роутинг заявок (AI)
- Подключение внешней базы знаний
- Отчёты и дашборды аналитики

### Этап 3

- Чат-бот для клиентов (портал, затем Telegram)
- AI-аналитика на естественном языке
- Расширенная CMDB для облачных ресурсов (1С, RDP, виртуальные машины)

### Далее

- Новые требования добавляются по мере поступления от заказчика

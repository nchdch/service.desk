# Подпроект 1: Frontend-фундамент — дизайн

**Дата:** 2026-06-14
**Статус:** утверждён, готов к написанию плана реализации

## Контекст

Второй подпроект реализации корпоративного портала «Виртуальный офис SD» (см. [docs/TZ.md](../../TZ.md)), следует за [подпроектом 0 (Фундамент)](2026-06-13-foundation-design.md) — готовый и задеплоенный backend (NestJS + Prisma + PostgreSQL, JWT-аутентификация, RBAC, модели `Organization`/`User`, 4 тестовые роли).

Ранее в дизайне подпроекта 0 предполагалось, что фронтенд и портирование дизайн-системы целиком войдут в «подпроект 1 (Service Desk)». По факту это слишком большая единица работы — она дальше декомпозирована на:

1. **Frontend-фундамент** (этот документ) — техническая база фронтенда: Next.js, перенос дизайн-системы, аутентификация, базовый layout, админ-разделы на уже существующем API.
2. Модуль заявок (backend): `Ticket`/`Comment`/`TimeEntry`/SLA + CRUD API + RBAC.
3. «Рабочее место инженера» (frontend) — экран по готовому дизайну из `design/` на данных подпроекта 2.
4. AI-помощник инженера — интеграция AI-панели с Claude API.

В рабочей копии репозитория уже есть незакоммиченный пакет `design/` — hi-fi дизайн + переносимая дизайн-система для экрана «Рабочее место инженера» (используется частично, см. ниже).

**Вне объёма этого подпроекта:**
- Экран «Рабочее место инженера», AI-панель — подпроекты 3-4.
- Любые сущности и API, связанные с заявками/оборудованием/базой знаний.
- Self-registration, восстановление пароля — пользователи только сидируются администратором (как в подпроекте 0).
- Docker — как и для backend, не используется; всё разворачивается нативно на сервере 10.3.0.88.

## Технический подход

- **Next.js (App Router) + TypeScript** — актуальный стандарт Next.js, соответствует ТЗ (раздел 5).
- **Сессия — httpOnly cookie + BFF-паттерн**: Next.js Route Handlers вызывают NestJS backend server-to-server и сами управляют cookie с JWT. Токен никогда не попадает в браузерный JS (требование ТЗ §4 — защита персональных данных). Реальная авторизация (RBAC) остаётся на backend (guards подпроекта 0) — фронтенд только маршрутизирует и скрывает недоступные пункты UI.
- **Дизайн-система** — переносится «как есть» (`styles.css`, `tokens/`, шрифты, лого) из `design/design-system/`, подключается глобально. Компоненты портируются из `.jsx` в `.tsx` по мере необходимости.
- **Менеджер пакетов** — npm (как в `backend/`).

## Auth-флоу

1. `/login` — публичный маршрут (доступен без cookie). Форма: email, пароль.
2. Форма отправляет POST на `/api/auth/login` (Route Handler).
3. Route handler вызывает `POST {BACKEND_URL}/auth/login` (backend на `http://localhost:3000` — server-to-server, наружу backend не открыт).
4. При успехе backend возвращает `{ accessToken, user }`. `accessToken` кладётся в cookie `session`: `httpOnly`, `Secure` (в проде), `SameSite=Lax`, `maxAge` = `SESSION_MAX_AGE_SECONDS` (значение из `.env` фронтенда, согласовано с `JWT_EXPIRES_IN` backend). Ответ — редирект на `/`.
5. При ошибке (401 от backend) — JSON `{ error: "Неверный email или пароль" }`, форма показывает текст под полями.
6. `middleware.ts`: все маршруты, кроме `/login` и `/api/auth/*`, требуют наличия cookie `session`. Нет cookie → редирект на `/login`. Проверяется только наличие, не подпись — реальная проверка валидности токена происходит на backend при любом запросе.
7. Серверные компоненты вызывают `getCurrentUser()` из `lib/auth.ts` → `GET /auth/me` с `Authorization: Bearer <session>`. Если backend вернёт 401 (истёкший/невалидный токен) — `lib/api.ts` инициирует редирект на `/login` с очисткой cookie.
8. Logout: кнопка в sidebar → POST `/api/auth/logout` → очищает cookie `session` → редирект на `/login`.

## Layout и навигация

**Sidebar (248px, токены дизайн-системы)**:
- Шапка бренда: логотип (`design-system/assets/brand/logo-mark.png`) + «Виртуальный офис» / «SERVICE DESK».
- Навигация, группы с overline-заголовками:
  - **«РАБОТА»** (видна всем ролям): Заявки, Мои заявки, Оборудование, База знаний, Аналитика — с Badge «скоро», ведут на общую страницу-заглушку «Раздел в разработке».
  - **«УПРАВЛЕНИЕ»** (только ADMIN): Пользователи, Организации. Для остальных ролей секция скрыта целиком.
- Низ: карточка пользователя (`Avatar` с инициалами, имя, роль) + кнопка «Выйти».

**Topbar (52px)** над контентной областью — заголовок текущей страницы.

Видимость пунктов навигации определяется ролью из `getCurrentUser()` на сервере (в layout `(portal)`).

## Страницы

1. **Главная** (`/`, все роли) — карточка приветствия: имя, email, `Badge` с ролью, название организации (если есть — см. backend-доработки ниже).
2. **Пользователи** (`/users`, ADMIN) — `DataTable`: Имя, Email, Роль (`Badge`), Организация, Статус активности (`Badge`), данные из `GET /users`.
3. **Организации** (`/organizations`, ADMIN) — `DataTable`: Название, Дата создания, данные из `GET /organizations`.

Прямой заход не-ADMIN на `/users` или `/organizations` (URL руками) → backend вернёт 403 → страница «Доступ запрещён» (без редиректа, т.к. пользователь аутентифицирован, просто не имеет прав).

## Backend-доработки (в рамках этого подпроекта)

Без них страницы выше показывали бы «голые» UUID вместо названий организаций — точечные, обратно совместимые добавления полей:

- **`UsersService.findAllSafe()`** — добавить `organization: { select: { name: true } }` через Prisma `include`/`select`, в ответе API — плоское поле `organizationName: string | null`.
- **`GET /auth/me`** — `AuthController.me()` становится асинхронным: при наличии `organizationId` резолвит `organizationName` через `OrganizationsService`/Prisma; итоговый ответ — `AuthenticatedUser` + `organizationName: string | null`.
- Обновить существующие e2e-тесты backend (`auth.e2e-spec.ts`), которые проверяют состав ответа `/auth/me` и `/users`, под новое поле.

  - **DataTable** — для страниц «Пользователи» и «Организации».

## Обработка ошибок

- Неверный email/пароль → 401 от backend → «Неверный email или пароль» под формой логина.
- Истёкший/невалидный токен (401 на `/auth/me` или данных страниц) → редирект на `/login`, cookie `session` очищается.
- 403 (не-ADMIN на `/users`/`/organizations`) → страница «Доступ запрещён».
- Backend недоступен / 5xx → `app/error.tsx` («Что-то пошло не так, попробуйте позже»).
- Пустые списки (`GET /users`/`GET /organizations` вернули `[]`) → текст «Список пуст» в `DataTable`.

## Тестирование

- **Компонентные тесты** — Vitest + React Testing Library: форма логина (успех/ошибка), `DataTable` (рендер строк, пустой список), фильтрация пунктов навигации по роли.
- **E2E** — Playwright, один сквозной сценарий на реальном стеке (frontend на 3001 + backend на 3000, сервер 10.3.0.88): логин под каждой из 4 тестовых ролей → проверка видимых пунктов навигации → для ADMIN — переход на `/users` и `/organizations`, проверка данных → logout.
- Состав/глубину тестов можно скорректировать на этапе плана реализации, если набор окажется избыточным для фундамента.

## Окружение разработки и деплоя

- Сервер 10.3.0.88 (`vrtvs01`, Ubuntu 24.04, доступ по SSH-ключу) — то же окружение, что и для backend. Node.js 20 LTS уже установлен.
- `frontend/` — сиблинг `backend/` в `~/service.desk`. Код редактируется локально, синхронизация через git (как для backend).
- `frontend/.env` (на сервере, не в git): `BACKEND_URL=http://localhost:3000`, `SESSION_MAX_AGE_SECONDS`, `NODE_ENV`; в репозитории — `frontend/.env.example`.
- Backend наружу из браузера не открыт — все обращения идут через Next.js (порт 3001), который ходит на backend по `localhost:3000`. CORS на backend не требуется.
- Запуск: `cd frontend && npm install` → dev: `npm run dev` (порт 3001); прод-запуск (process manager — pm2/systemd либо `nohup`, аналогично текущему запуску backend) уточняется в плане реализации.

## Критерии готовности (definition of done)

1. `frontend/` развёрнут на сервере 10.3.0.88, доступен на порту 3001.
2. Незалогиненный пользователь на любом защищённом маршруте редиректится на `/login`.
3. Логин под каждой из 4 тестовых ролей работает; неверный пароль — корректное сообщение об ошибке.
4. После логина: «Главная» показывает имя/email/роль (и организацию — для CLIENT); sidebar показывает корректный набор пунктов для роли (ADMIN видит «Пользователи»/«Организации», остальные — нет).
5. ADMIN: страницы «Пользователи» и «Организации» показывают реальные данные из backend (включая названия организаций); прямой заход других ролей на эти URL → «Доступ запрещён».
6. Logout очищает сессию и возвращает на `/login`.
7. Дизайн-система и токены перенесены и используются (визуально соответствует `design/`, где применимо); шрифты Golos Text/JetBrains Mono подключены.
8. Компонентные и e2e-тесты зелёные.
